## 1. רקע ומטרת הפרויקט

בהתאם לעקרונות שנלמדו בהרצאה L06 (MCP & API for Agents), פרויקט זה בונה סוכן AI שמתחבר לשני שירותי Google - Gmail ו-Calendar - ומבצע תהליך ניהול הזמנות פגישה מקצה לקצה: קריאת מיילים אחרונים, זיהוי אילו מהם מכילים בקשה חופשית לתיאום פגישה, חילוץ פרטי הפגישה, בדיקת זמינות ביומן, ותגובה אוטומטית מתאימה - אישור (עם יצירת אירוע ביומן) או דחייה מנומסת (עם בקשה להצעת מועד חלופי).

הפרויקט משתמש בגישת Skill+API (רמה 2 מתוך שלוש הרמות המתוארות בהרצאה: API ישיר / Skill / MCP מלא) - כל שירות Google נגיש דרך קוד Python ישיר (לא MCP), בעוד שההחלטות הדורשות הבנת שפה טבעית (זיהוי הזמנה, חילוץ פרטים, ניסוח תגובה) מבוצעות על ידי Claude Code דרך Skills ייעודיים.

### זרימת העבודה

1. `gmail_client.py` קורא מיילים מהיומיים האחרונים, **תוך אי-הכללת מיילים שנשלחו על ידי הסוכן עצמו** (ראו סעיף 5.2 - סיבה חשובה)
2. עבור כל מייל: `scan-gmail` Skill מחליט YES/NO - האם זו בקשת פגישה אמיתית
3. אם כן: `parse-meeting-request` Skill מחלץ תאריך, שעה, משך, מיקום כ-JSON מובנה
4. `calendar_client.py` בודק זמינות אמיתית ביומן Google Calendar (Free/Busy API)
5. אם פנוי: נוצר אירוע אמיתי ביומן; `manage-calendar` Skill מנסח מייל אישור
6. אם תפוס: `manage-calendar` Skill מנסח מייל דחייה מנומס
7. `gmail_client.py` שולח את התגובה בפועל בחזרה לשולח המקורי

### מנגנון בטיחות: DRY RUN כברירת מחדל

כל פעולה בלתי הפיכה (יצירת אירוע ביומן, שליחת מייל) מוגנת תחת דגל `--send`. ללא הדגל, התוכנית מריצה DRY RUN מלא - מזהה, מחלצת, בודקת זמינות, ומנסחת תגובה - אך **לא** יוצרת אירוע ולא שולחת מייל. זה מאפשר בדיקה בטוחה וחוזרת בלי תופעות לוואי לא רצויות.

## 3. תהליך ההקמה (Google Cloud Setup)

לצורך שחזור, אלו השלבים המדויקים שבוצעו:

1. יצירת פרויקט חדש ב-Google Cloud Console
2. הפעלת Gmail API ו-Google Calendar API (APIs & Services -> Library)
3. הגדרת OAuth Consent Screen: User Type = External, מצב Testing
4. הוספת משתמשי בדיקה (Test users) - כל חברי הקבוצה
5. יצירת OAuth Client ID מסוג Desktop app, הורדת קובץ client_secret.json
6. הרצת `authenticate.py` פעם אחת - פותח דפדפן, מבקש התחברות ואישור הרשאות (Gmail read/send, Calendar), ויוצר `token.json` לשימוש חוזר

### אבטחת מידע

בהתאם לעקרונות שנלמדו בהרצאה (סעיף 3.4): `client_secret.json` ו-`token.json` נשמרים בתיקיית `credentials/` שנמצאת ב-`.gitignore` מהרגע הראשון, לפני כל commit. אף מפתח, טוקן, או סוד לא נכנס להיסטוריית ה-git בשום שלב.

## 4. תרחישי בדיקה ותוצאות

### תרחיש 1: זיהוי שלילי נכון (No False Positives)

הרצה על 10 מיילים אמיתיים מהתיבה (ניוזלטרים, פרסומות, התראות משרות, חיובי אשראי) - **אף אחד מהם לא זוהה בטעות כהזמנת פגישה**, כולל מייל עם נושא מטעה כמו "You have 1 new invitation" (הזמנה גנרית מפלטפורמה חיצונית, לא בקשה אישית).

### תרחיש 2: זיהוי חיובי + מסלול "פנוי" (יצירת אירוע)

חבר/ת קבוצה שלח/ה מייל בדיקה בטקסט חופשי: "אפשר להיפגש ביום שישי הקרוב בשעה 23:00 בקפיטריה לדבר על התרגיל?"

תוצאה:
- זוהה כהזמנת פגישה: כן
- פרטים שחולצו: תאריך 2026-07-17, שעה 23:00, מיקום קפיטריה
- זמינות ביומן: פנוי
- אירוע נוצר ביומן בפועל (Google Calendar Event ID: asknfj58sab37mc5qi1k6fq9n4)
- מייל אישור נשלח בפועל לשולח

### תרחיש 3: מסלול "תפוס" (דחייה)

מייל בדיקה נוסף לאותו יום/שעה תפוסה - הסוכן זיהה נכון שהיומן תפוס, ושלח מייל דחייה מנומס בעברית תקינה, ללא הצעת תאריך חלופי מומצא (בהתאם לכלל ב-Skill).

## 5. ניתוח וממצאים - שלוש תגליות הנדסיות

### 5.1 חוסר התאמה בין פרויקטי Google Cloud

בשלב ההקמה נתקלנו בשגיאת `403: access_denied` עקבית, למרות שמשתמש הבדיקה נראה מוגדר נכון. הסיבה: ה-OAuth Client שנוצר היה שייך לפרויקט Google Cloud שונה (`groovy-reserve-...`, פרויקט ברירת מחדל שנוצר אוטומטית) מהפרויקט שבו הוגדרו ה-consent screen וה-test users (`biu-len05-agent`). Google Cloud מנהל הרשאות **פר-פרויקט**, ולכן שני חלקי ההגדרה חייבים לחיות באותו פרויקט בדיוק. הפתרון: יצירת Client חדש בתוך הפרויקט הנכון, ווידוא ששני ה-APIs מופעלים גם הם שם.

זהו יישום מעשי ישיר של מה שההרצאה מכנה "Trust Provider" (סעיף 3.2) - שרשרת אמון שדורשת עקביות בכל שלב.

### 5.2 לולאת משוב מתגובות עצמיות (Feedback Loop)

בהרצה מוקדמת, המערכת שלחה מיילי תגובה אמיתיים - שהם עצמם נכנסו לתיבה כמיילים חדשים. בהרצה הבאה (עדיין בטווח `days_back=2`), המערכת **זיהתה את התגובות של עצמה כהזמנות פגישה חדשות** וניסתה להגיב להן שוב, מה שיצר אירועי יומן כפולים ומיילים מיותרים.

הפתרון: הוספת `-from:me` לשאילתת החיפוש ב-Gmail API, המסנן בצד השרת כל מייל שנשלח מהחשבון של המשתמש עצמו, לפני שהתוצאות מגיעות לקוד כלל.

זו דוגמה מוחשית לסיכון שההרצאה מרמזת עליו בעקיפין (ניהול תהליכים אוטונומיים, סעיף 4.6) - מערכת הפועלת ללא זיכרון (state) של פעולות קודמות עלולה ליצור לולאות בלתי צפויות כשהיא קוראת וכותבת לאותו ערוץ.

### 5.3 אי-דטרמיניזם של החלטות LLM בינאריות

באחד המבחנים, אותו מייל בדיוק הורץ פעמיים ברצף דרך אותו קוד (`python main.py --only <id>`) - בפעם הראשונה `scan-gmail` החזיר `NO` (לא הזמנת פגישה), ובפעם השנייה - על אותו קלט זהה לחלוטין - החזיר `YES`.

זהו ממצא חשוב: **החלטות סיווג בינאריות מבוססות LLM אינן דטרמיניסטיות מובנֵית**, גם כשהקלט זהה במדויק. בנוסף, נצפו מספר timeouts (חריגה מ-120 שניות) בקריאות רצופות מהירות ל-Claude Code CLI, כולל על מיילים שבהרצות אחרות זוהו תוך שניות - מה שמרמז על רגישות לעומס בצד השרת בזמני תגובה, לא רק על תוכן הקלט.

**המשמעות המעשית:** מערכת production אמיתית המבצעת פעולות בלתי הפיכות (כמו יצירת אירוע ביומן) על בסיס סיווג LLM בודד, חשופה לסיכון של החלטות לא עקביות. פתרונות אפשריים (שלא מומשו בפרויקט זה, אך מוזכרים כהמלצה): `temperature=0` להפחתת אקראיות, מנגנון הצבעת רוב (self-consistency - הרצת אותה שאלה מספר פעמים ובחירת התשובה השכיחה), ואימות אנושי לפני פעולות קריטיות.

## 6. מגבלות ידועות

- **אי-אידמפוטנטיות**: אין מנגנון לסימון מיילים שכבר טופלו (למשל תיוג Gmail label) - הרצה חוזרת על אותו טווח זמן עלולה לעבד מחדש את אותם מיילים
- **מצב Testing ב-OAuth**: מוגבל ל-100 משתמשי בדיקה לכל אורך חיי האפליקציה; ה-token עשוי לדרוש רענון תקופתי
- **דיוק הזיהוי לא נמדד בקנה מידה גדול**: המדגם קטן (מספר מיילי בדיקה בודדים), כפי שהודגם בסעיף 5.3, יש גם שונות (variance) מובנית שדורשת מדגם גדול יותר להערכה סטטיסטית אמינה
- **ללא MCP מלא**: הפרויקט משתמש בגישת Skill+API ישירה (רמה 2), לא בפרוטוקול MCP עצמו (רמה 3) - הרחבה עתידית אפשרית

## 7. הפעלה עצמאית

### דרישות מקדימות
- Python 3.10+, Claude Code מותקן
- פרויקט Google Cloud עם Gmail API + Calendar API מופעלים, OAuth Client (Desktop app)
- חבילות: `google-api-python-client`, `google-auth-httplib2`, `google-auth-oauthlib`

### התקנה ואימות ראשוני

    python -m venv venv
    venv\Scripts\Activate.ps1
    pip install google-api-python-client google-auth-httplib2 google-auth-oauthlib
    python src\authenticate.py

### הרצה

    cd src
    python main.py                          # DRY RUN על מיילים אחרונים
    python main.py --send                    # LIVE - פעולות אמיתיות
    python main.py --only <message_id>        # מייל בודד, DRY RUN
    python main.py --only <message_id> --send  # מייל בודד, LIVE
